# nano-vLLM 学习期 TODO 清单

> 目标：先记录“做什么、为什么做、做到什么算完成”，暂不进入实现细节。随着代码学习推进，持续补充和重排优先级。

## 使用规则
- 每次新增任务只写三件事：`目标`、`最小需求`、`完成标准`。
- 学习没覆盖到的模块，不做过早设计。
- 优先保证兼容现有接口行为，避免破坏当前可用能力。

## P0（当前首要）
- [ ] 在线接口 + 完整 Continuous Batching
  - 目标：让请求可以持续进入系统，不必等一整批结束再返回。
  - 最小需求：
    - 支持请求动态入队（运行中可继续加请求）。
    - 支持流式返回（按 token 或按片段返回）。
    - 采用统一 token 预算调度思路，弱化“prefill/decode 完全分离”的硬边界。
    - 保留现有离线 `generate` 行为（兼容旧用法）。
  - 完成标准：
    - 在多请求并发下，系统可持续调度且无明显阻塞。
    - 单请求与多请求场景都能正确结束并回收 KV 资源。

- [ ] Chunked Prefill（分块预填充）
  - 目标：长 prompt 不再一次性吃满预算，避免阻塞 decode 与新请求准入。
  - 最小需求：
    - 支持将单个长 prompt 切分为多轮 prefill chunk。
    - chunk 与 decode 可在统一调度中共存推进。
    - 不破坏现有 KV cache 与 prefix cache 语义。
  - 完成标准：
    - 长短请求混部时，短请求首 token 延迟明显改善。
    - 与现有结果保持一致（同采样参数下输出行为不异常）。

- [ ] KV Cache 分配与管理策略对齐 vLLM v1（PagedAttention 路线）
  - 目标：提升显存弹性，增强在线场景稳定性。
  - 最小需求：
    - 统一 block/page 抽象。
    - 可扩展的分配/回收策略与可观测指标。
    - 明确“物理显存预留”与“逻辑块回收复用”的边界语义，避免误把回收理解为频繁 cuda free。
    - 不破坏现有 prefix cache 语义。
  - 完成标准：
    - 长会话与并发混部场景下，显存行为更稳定。
    - 分配/回收路径可观测且可诊断。

## P1（学习完成后优先开发）
- [ ] 请求取消与超时控制
  - 目标：支持在线服务常见中断场景。
- [ ] 基础服务化封装（HTTP/gRPC 二选一）
  - 目标：把引擎能力暴露为稳定服务接口。
- [ ] 观测能力（日志、吞吐、延迟、队列长度）
  - 目标：能定位性能瓶颈和调度问题。

## P2（向 vLLM v1 能力靠拢）

- [ ] 更完善的调度策略（公平性、优先级、饥饿保护）
- [ ] Prefix Cache 策略增强（命中率、淘汰策略、可观测性）
- [ ] 多模型/多实例管理能力（按资源做隔离）

## P3（工程化完善）

- [ ] 配置体系与启动参数规范化
- [ ] 文档补全（架构图、调用链、故障排查）

## 里程碑
